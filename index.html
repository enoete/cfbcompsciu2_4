üë• Key Stakeholders

Case Study 2: NextGen Social - A New Social Media Platform

Moderate Level ‚è±Ô∏è 25 minutes

üìã The Scenario

Client: NextGen Social, a startup with $2M in seed funding

Project: Build a social media platform that combines elements of Instagram, TikTok, and Discord, targeting Gen Z users (ages 13-24)

Current Situation: The founders have a vision but aren't entirely sure what features will resonate with users. They want to launch fast to beat competitors and iterate based on user feedback. They've allocated 6 months to MVP (Minimum Viable Product).

üë• Key Stakeholders

  • CEO/Founder: "We need to fail fast and learn fast. What teens want today might change tomorrow."
  • Lead Developer: "We should use proven technology stacks and avoid reinventing the wheel."
  • UX Designer: "User experience is everything. If Gen Z doesn't love it immediately, they'll delete it."
  • Investors: "Show us user growth and engagement metrics within 6 months, or funding stops."

‚ö†Ô∏è Constraints & Challenges

  • Limited budget: $2M must cover development, servers, and initial marketing
  • Small team: 4 developers, 1 designer, 1 product manager
  • Uncertain requirements: Target users haven't clearly articulated what they want
  • Fast-moving market: Competitor apps launch new features weekly
  • Need to reach 100,000 users within 6 months to secure Series A funding

ü§î Discussion Questions

Question 1: Which SDLC model would you choose for this project? Consider: Uncertain requirements, need for speed, user feedback importance, and market volatility. Would Waterfall work here? What about Agile or Prototyping model?
Question 2: How would you handle requirements elicitation when your target users are teenagers who may not know what they want? Think about: Traditional techniques vs. creative approaches. Would interviews work? Questionnaires? What about observational studies of how teens use current social media?
Question 3: The CEO wants to "move fast and break things." How do you balance speed with quality? Consider: MVP approach, technical debt, scalability concerns, and user trust. What gets built in Phase 1 vs. Phase 2?
Question 4: Should you reuse existing components or build everything custom? Analyze: Authentication systems, video processing, messaging infrastructure, content recommendation algorithms. When does "buy vs. build" make sense?

üìù Your MVP Decision & Justification

üéØ Discussion Points for Instructors

SDLC Model: Agile (Scrum or Kanban) is ideal here because:

  • Requirements are unclear and will evolve based on user feedback
  • Speed to market is critical in the competitive social media space
  • 2-week sprints allow rapid iteration and course correction
  • Early user testing helps validate or pivot features quickly

Prototyping Model could also work for initial requirements gathering - build quick UI mockups to test with Gen Z focus groups before committing to development.

Requirements Elicitation: Prototyping + observations are key! Traditional interviews may not work well with teens. Consider: creating mockups/wireframes, observing how they use existing apps, beta testing with small user groups, and A/B testing different features.

Component Reuse: Strongly recommended! Use existing libraries for: authentication (Auth0, Firebase), video processing (AWS MediaConvert), messaging (Twilio, Stream), and content delivery (CloudFront). This saves months of development time.

Typical MVP: User profiles, photo/video sharing, direct messaging, and basic feed. Save AI recommendations, live streaming, and advanced filters for later iterations based on what users actually want.

Case Study 3: MedAlert - Emergency Medical Dispatch System

Advanced Level ‚è±Ô∏è 30 minutes

üìã The Scenario

Client: City of Riverside Emergency Services Department

Project: Develop a mission-critical dispatch system that coordinates ambulances, fire trucks, and police during emergencies. The system must integrate with 911 call centers, hospital systems, GPS tracking, and traffic data.

Current Situation: Using a 20-year-old system that frequently crashes. Response times have increased by 30% in the past year. Lives are at stake. The city council has approved a $10M emergency budget after a recent incident where system failure contributed to a preventable death.

üë• Key Stakeholders (With Conflicting Priorities!)

  • Fire Chief: "We need real-time GPS tracking of all units and integration with building blueprints for fire responses."
  • Police Chief: "Security is paramount. We can't risk criminals accessing our unit locations. Also need integration with criminal databases."
  • Hospital Network Director: "We need advance notice of incoming patients with their vitals and condition. Every second counts in trauma."
  • 911 Call Center Manager: "Our operators need a system that's intuitive under extreme stress. Current system requires 40 hours of training and still confuses people."
  • City Council: "This better work perfectly. We can't afford another failure. Also, we need to see results within 12 months for the election cycle."
  • IT Security Officer: "This is a critical infrastructure system. We need bulletproof cybersecurity, redundancy, and disaster recovery."

‚ö†Ô∏è Extreme Constraints & Risks

  • Zero tolerance for failure: System downtime literally costs lives
  • Political pressure: 12-month timeline is aggressive for this complexity
  • Integration nightmare: Must integrate with 30+ legacy systems, many with poor documentation
  • High stakes: Legal liability if system failure leads to death or injury
  • 24/7 operations: Cannot take system offline for updates or maintenance
  • Regulatory compliance: HIPAA (healthcare data), CJIS (criminal justice), and FCC (emergency communications)
  • User stress: System must work flawlessly under high-stress emergency conditions

ü§î Complex Discussion Questions

Question 1: This project has characteristics that could support MULTIPLE SDLC models. What's the best approach? Consider: Could you use Waterfall for the critical core dispatch logic (where reliability is paramount) but Agile for the user interface? Is a hybrid approach feasible? What about the Spiral model given the extreme risks?
Question 2: How do you handle conflicting stakeholder requirements? Example conflict: Police want maximum security (may slow system down), while Fire Chief wants speed above all else (may sacrifice some security). Who wins? How do you negotiate?
Question 3: The timeline is aggressive. How do you determine what's feasible? Conduct a feasibility analysis considering: Technical feasibility (can we integrate 30+ systems in 12 months?), Economic feasibility (is $10M enough?), Operational feasibility (can we train 200+ dispatchers?), and Schedule feasibility (12 months realistic?).
Question 4: How would you approach requirements elicitation for users who work in high-stress environments? 911 operators and dispatchers work under extreme pressure. Traditional interviews might not capture the real requirements. What techniques would work better?
Question 5: This is a mission-critical system. What safeguards must be in place? Think about: Redundancy, failover systems, testing strategies, deployment approach, training plans, and rollback procedures if something goes wrong.

üî• Bonus Challenge: The Spiral Model Application

  1. Spiral 1 (Months 1-3): What would you build and validate?
  2. Spiral 2 (Months 4-6): What gets added? What risks are addressed?
  3. Spiral 3 (Months 7-9): How do you expand functionality?
  4. Spiral 4 (Months 10-12): Final integration and deployment

For each spiral, identify: Main objectives, risks to address, prototype/deliverable, and validation method.

This project highlights that there's rarely a "perfect" SDLC model for complex projects. Write your reflection:

üìù Your Analysis & Recommendations

üéØ Discussion Points for Instructors

Best Approach: Spiral Model (or Hybrid Spiral-V Model)

Rationale:

  • High Risk: Lives are at stake, making systematic risk analysis essential in each iteration
  • Complex Integration: 30+ legacy systems mean you need incremental integration with validation at each step
  • Regulatory Requirements: Multiple compliance frameworks require thorough documentation and validation
  • Stakeholder Conflicts: Multiple iterations allow stakeholders to see prototypes and negotiate trade-offs

Sample Spiral Iteration Plan:

Spiral 1: Core dispatch system with basic GPS tracking. Risk focus: Can we achieve required response time? Prototype tested with small group of dispatchers.

Spiral 2: Add hospital integration and patient data. Risk focus: HIPAA compliance and data security. Validate with healthcare partners.

Spiral 3: Integrate criminal databases and building blueprints. Risk focus: Integration with legacy systems. Test with police and fire departments.

Spiral 4: Full system integration, redundancy, and deployment. Risk focus: Failover procedures and training effectiveness.

Requirements Elicitation: Use observations (ride-alongs with dispatchers and emergency responders), simulated emergency scenarios (stress testing the UI), interviews during calm periods, and prototyping with iterative feedback. Avoid pure questionnaires - they won't capture the stress and complexity of real emergencies.

Stakeholder Conflicts: Establish a priority framework: 1) Life safety (reliability and speed), 2) Security (protect sensitive data), 3) Usability, 4) Integration breadth. Use this framework to make trade-off decisions transparently.

Why NOT Agile? While Agile offers flexibility, this project requires extensive upfront planning, rigorous testing, and regulatory compliance documentation. However, Agile principles (iterative development, user feedback) can be incorporated within the Spiral framework.

Feasibility Concerns: 12 months is extremely aggressive for this complexity. A realistic timeline would be 18-24 months. If forced to meet 12 months, recommend phased deployment: Core system in 12 months, advanced features in subsequent phases.

Case Study 4: EduFlex - University Learning Management System

Moderate Level ‚è±Ô∏è 25 minutes

üìã The Scenario

Client: State University with 35,000 students and 2,500 faculty members

Project: Replace their current Learning Management System (LMS) which is outdated and unpopular. Students complain it's clunky, professors say it's missing key features, and IT says it doesn't integrate well with other campus systems.

Decision Point: Should they build a custom system or buy/customize an existing platform (like Canvas, Blackboard, or Moodle)?

üë• Key Stakeholders

  • Students: "Make it work like TikTok - simple and intuitive. We want mobile access and video support."
  • Faculty: "We need robust grading tools, plagiarism detection, and easy content creation. Also, we've spent years building content in the old system."
  • IT Department: "Must integrate with our student information system, email, library, and authentication system."
  • Administration: "We need analytics on student performance and retention. Budget is limited."
  • Accessibility Office: "Must be fully accessible for students with disabilities (ADA/WCAG 2.1 compliant)."

‚ö†Ô∏è Constraints & Considerations

  • Must be ready for Fall semester (9 months from now)
  • Data migration: 10 years of courses, assignments, grades, and student work
  • Faculty resistance to change: Many professors are not tech-savvy
  • 24/7 availability required (online students in different time zones)
  • Must support 35,000 concurrent users during peak times (finals week)
  • Budget: $1.5M (tight for custom development, but possible for commercial license)

ü§î Discussion Questions

Question 1: Build vs. Buy - What factors should influence this decision? Consider: Cost, timeline, customization needs, maintenance burden, vendor lock-in, and technical capabilities of IT department.
Question 2: If building custom, which SDLC model fits best? Consider: Known requirements from current system pain points, but also need for user feedback to ensure adoption. Incremental? Agile? Spiral?
Question 3: If buying commercial software, how do requirements engineering principles still apply? Think about: Requirements gathering influences which product to buy, customization needs, integration requirements, and training plans.
Question 4: How do you ensure faculty adoption? Challenge: Even the best system fails if users resist it. What strategies encourage buy-in from tech-resistant professors?
Question 5: What requirements elicitation techniques work best for gathering input from 35,000 students? You can't interview everyone. How do you get representative feedback efficiently?

üìù Your Build vs. Buy Decision

üéØ Discussion Points for Instructors

Most Realistic Recommendation: Buy Commercial Software

Rationale:

  • Timeline: 9 months is insufficient for custom development of a robust LMS
  • Budget: $1.5M is barely enough for development, let alone ongoing maintenance
  • Proven Solution: Commercial platforms have been refined over years with thousands of schools' feedback
  • Maintenance: Universities typically lack the resources to maintain complex custom systems
  • Updates & Security: Vendors handle security patches and feature updates

When Custom Makes Sense: Only if the university has unique pedagogical requirements that no commercial system supports AND has a strong, dedicated development team AND has a longer timeline (18+ months).

Requirements Engineering Still Critical: Even when buying, you must:

  • Gather requirements to evaluate which product best fits needs
  • Identify integration requirements with existing systems
  • Determine customization needs and priority
  • Plan data migration requirements
  • Define training and support requirements

If Building Custom - Use Incremental Model: Deploy core features (course creation, gradebooks, assignments) in Phase 1, then add advanced features (video integration, analytics) in Phase 2+. This allows faster time-to-market while incorporating user feedback.

Requirements Elicitation: Questionnaires for students (large population), focused interviews with representative faculty from different disciplines, usability testing with prototypes, and review of pain points from current system's help desk tickets.

Faculty Adoption Strategies: Early faculty advisory group, pilot program with early adopters, comprehensive training program, dedicated support during transition, and migration of existing content (don't force them to rebuild from scratch).

Case Study 5: HomeMind - AI-Powered Smart Home Ecosystem

Advanced Level ‚è±Ô∏è 25 minutes

üìã The Scenario

Client: HomeMind Technologies, a mid-sized company entering the competitive smart home market

Project: Develop an integrated smart home system with AI capabilities that learns user behavior and automates home functions (lighting, temperature, security, entertainment). Must compete with established players like Google Home, Amazon Alexa, and Apple HomeKit.

Complexity Factors: Hardware + software integration, real-time processing, AI/ML components, privacy concerns, and interoperability with hundreds of third-party devices.

üë• Key Stakeholders

  • Product Manager: "Our differentiator is AI that truly learns. Not just simple automation rules - actual machine learning that predicts what users want."
  • Hardware Engineer: "We're designing custom sensors and a central hub. Firmware and software must be in perfect sync."
  • Security Expert: "We're entering people's homes - privacy and security are non-negotiable. One data breach destroys the company."
  • UX Designer: "Setup must be grandma-friendly. If installation requires an IT degree, we lose 80% of our market."
  • Marketing: "We need to launch at CES (Consumer Electronics Show) in 10 months to make a splash and secure retail partnerships."

‚ö†Ô∏è Project Challenges

  • Hardware-Software Dependency: Can't fully test software until hardware prototypes exist
  • AI Uncertainty: Machine learning behavior is somewhat unpredictable - how do you test it?
  • Ecosystem Integration: Must work with 100+ different smart bulbs, thermostats, cameras, etc.
  • Real-time Requirements: Security features must respond instantly to threats
  • Regulatory Compliance: FCC (radio frequency), UL (electrical safety), privacy laws (GDPR, CCPA)
  • Firmware Updates: Must support over-the-air updates for thousands of deployed devices

ü§î Multi-Layered Discussion Questions

Question 1: This is a hardware-software co-development project. How does this change your SDLC approach? Consider: Hardware has longer development cycles and is expensive to change. Software is more flexible. How do you synchronize these different timelines? Can you use different SDLC models for hardware vs. software?
Question 2: AI/ML introduces uncertainty. How do you handle requirements for a system that "learns"? Challenge: You can't fully specify the behavior of a machine learning system upfront. It evolves with data. How do you write requirements for this? How do you test it?
Question 3: This project involves THREE major components with different characteristics. How do you manage them?
  • Component A - Hardware (hub, sensors): Long dev cycle, expensive to change, physical prototyping needed
  • Component B - Core Software (device communication, automation logic): Critical reliability, must be stable
  • Component C - AI/ML (behavior learning): Experimental, will evolve based on user data

Should you use the same SDLC model for all three? Or different models for each component?

Question 4: How do you handle requirements gathering for a product that will compete with market leaders? Your competitors (Amazon, Google) set user expectations. Do you copy their features? Innovate? How do you elicit requirements when users have preconceived notions from competitor products?
Question 5: Security and privacy requirements are critical. How do you ensure they're not an afterthought? Many IoT products have been hacked. How do you build security into every phase rather than adding it at the end?

üìù Your Hybrid SDLC Strategy

üéØ Discussion Points for Instructors

Recommended Approach: Hybrid Model with Component-Specific Strategies

Hardware Component: Use V-Model or Waterfall

  • Hardware changes are expensive; extensive upfront design is essential
  • V-Model ensures each design phase has corresponding testing/validation
  • Physical prototyping happens in stages (breadboard ‚Üí PCB prototype ‚Üí final design)

Core Platform: Use Incremental Model with Spiral Elements

  • Build critical features (networking, security) first with rigorous testing
  • Spiral approach for high-risk components (encryption, device communication)
  • Incremental releases allow testing with hardware prototypes as they become available

AI/ML Component: Use Prototyping + Agile

  • ML models require experimentation - build, test, iterate rapidly
  • Start with simulated data, then real user data in beta testing
  • Continuous refinement post-launch as system learns

Mobile App: Use Agile (Scrum)

  • User interface will evolve based on usability testing
  • 2-week sprints allow rapid iteration on UX
  • Can be developed somewhat independently from hardware

Integration Strategy: Establish "integration milestones" where all components come together for system testing. Use continuous integration tools to catch compatibility issues early.

AI Testing Approach:

  • Unit Tests: Test individual ML model components
  • Behavioral Tests: Define expected behavior patterns (e.g., "should learn user's wake-up time within 2 weeks")
  • A/B Testing: Deploy different models to different user groups, compare performance
  • Safety Tests: Ensure AI can't learn dangerous behaviors (e.g., ignoring security alerts)
  • Beta Program: Extensive real-world testing with volunteer users before general release

Security-First Approach: Use threat modeling (STRIDE framework) in the design phase, security code reviews at each sprint, penetration testing before each release, and automatic security updates post-launch.

Requirements Elicitation: Competitor analysis (understand baseline expectations), user studies with current smart home owners, usability labs with prototypes, beta tester feedback programs, and privacy focus groups (understand comfort levels with data collection).

Key Takeaway: Complex projects may require custom SDLC approaches that blend multiple methodologies. The key is being intentional about which model you use for each component and how they integrate together.
